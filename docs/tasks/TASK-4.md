## Контекст

Задачи живут в `docs/tasks/TASK-<N>.md` с YAML frontmatter между `---` ([стандарт frontmatter](../../.cursor/rules/docs-tasks/docs-tasks-frontmatter-standard.mdc), протокол в `.cursor/skills/task-management-protocol`). Нужен **MVP плагина** для наглядной работы с этими файлами из редактора.

Исходники расширения — в **корне этого репозитория** ([vscode-plugin-docs-tasks](https://github.com/taraswww777/vscode-plugin-docs-tasks)).

## Цели MVP

1. **Индекс задач**: обнаружить в открытом workspace все `docs/tasks/TASK-*.md`, распарсить frontmatter (ошибки парсинга показать в UI и не падать целиком).
2. **Таблица задач**:
   - колонки по полям frontmatter: минимум `id`, `title`, `status`, `created`, `updated`, при наличии — `type`, `tags`, `source.issue`;
   - **фильтрация** по произвольной комбинации полей (строка поиска по `title`, фильтр по `status`, по тегу и т.д. — конкретный набор правил реализации описать в README расширения);
   - **несколько сохранённых видов** (presets): пользователь задаёт имя, набор фильтров/сортировки/видимых колонок; можно открыть несколько таких видов (вкладки или отдельные панели — на усмотрение первой реализации, главное — N независимых конфигураций).
3. **Канбан**: колонки по статусам протокола `backlog | in-progress | done | cancelled`; карточка = задача; **drag-and-drop** между колонками записывает новый `status` и обновляет `updated` в файле.
4. **Редактор метаданных**: из контекста таблицы/канбана открыть форму (или боковую панель) для правки полей блока `---` с валидацией (валидные статусы, формат дат `YYYY-MM-DD`); сохранение без порчи остального markdown.

## Вне scope MVP (отдельные задачи позже)

- Диаграмма Ганта, **последовательные связи** между задачами, **группировка** в цепочки/иерархии — см. [TASK-5](./TASK-5.md).
- Синхронизация с GitHub Issues через API (достаточно ссылки `source.issue` и открытия в браузере по желанию).
- Новые поля frontmatter под даты начала/окончания и `depends_on` до согласования в отложенной задаче.

## Технические ориентиры

- TypeScript, официальный шаблон VS Code Extension; сложный UI — **Webview** или **WebviewView**.
- Парсинг/запись frontmatter: библиотека уровня `gray-matter` или эквивалент; кодировка UTF-8.
- Изменения на диске: `FileSystemWatcher` на `docs/tasks/**/*.md`, пересборка индекса с debounce.
- Webview: **CSP**, без небезопасного встраивания сырого markdown в HTML.
- Совместимость установки: **VS Code и Cursor** (локальный `.vsix`, инструкция в README пакета).

## Критерии приёмки MVP

1. Команды палитры (или entry в activity bar) открывают **таблицу** и **канбан**.
2. Таблица: сортировка, фильтры, **несколько сохранённых видов** на workspace (или глобально — зафиксировать в README).
3. Канбан: перетаскивание меняет `status` и `updated` в соответствующем `.md`.
4. Редактор метаданных: валидные правки `---`, ошибки показываются пользователю.
5. В корне репозитория есть **README**: как собрать, запустить в Extension Development Host, собрать `.vsix`.

## Сделано

Реализовано расширение **Docs Tasks** в корне репозитория: индекс + watcher, таблица (presets во `workspaceState`), канбан с DnD, редактор frontmatter (`gray-matter`, CSP во webview). Инструкции — [`README`](../../README.md).

## Следующие шаги (опционально)

- Persist последнего активного пресета (сейчас при открытии таблицы активна первая вкладка вида до первого сохранённого набора сообщений из webview).
- Тонкая синхронизация неизвестных ключей YAML при сохранении метаданных (сейчас сохранение перезаписывает описанный набор полей и сохраняет прочие ключи из распарсенного объекта там, где возможно через spread).
